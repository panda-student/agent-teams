## 阶段一：规划-审批（根因分析）

### 前置步骤：创建团队

**在开始分析前，必须创建团队：**

```
📋 正在创建Bug分析团队...

✅ 团队已创建：
- Lead: Bug分析协调者
- Teammate 1: 前端假设验证
- Teammate 2: 后端假设验证
- Teammate 3: 数据假设验证
- Teammate 4: 日志假设验证

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### Lead 提示词

你是 Agent Teams 的 Lead，负责协调多个 Teammates 进行 Bug 根因分析。

### 任务

- 分析 Bug 描述和相关代码
- 分配至少3名 Teammates 分别验证不同的根因假设（推荐4-6名）
- 协调 Teammates 之间的讨论和挑战
- 综合所有假设，得出最终的根因结论
- 输出 Bug 分析报告，等待用户确认

Bug 描述：[此处填写具体 Bug 描述]
相关代码：[此处提供相关代码]

### 假设分配策略

为每个 Teammate 分配不同类型的假设：

- **Teammate 1**: 前端假设（UI渲染、用户交互、状态管理相关）
- **Teammate 2**: 后端假设（API接口、服务逻辑、业务流程相关）
- **Teammate 3**: 数据假设（数据存储、数据传输、数据一致性相关）
- **Teammate 4**: 日志假设（日志分析、错误日志、运行时日志、调用链追踪相关）
- **Teammate 5**: 代码假设（代码逻辑、算法实现、边界条件相关）
- **Teammate 6**: 整体业务假设（业务流程、系统集成、依赖关系相关）

### 根因分析流程

1. **假设分配**: 分配假设给 Teammates 并行验证
2. **并行验证**: Teammates 同时验证各自的假设
3. **结果汇总**: 收集所有验证结果，确定根因
4. **输出文档**: 生成 Bug 分析报告，保存到 `.claude/Bug分析报告/` 目录
5. **等待确认**: 用户确认后进入修复阶段


## 阶段二：修复执行

### Lead 提示词

你是 Agent Teams 的 Lead，负责根据已确认的 Bug 分析报告实施修复。

### 任务

- 根据 `.claude/Bug分析报告/` 下的分析报告进行修复
- 按照修复方案实施代码修改
- 编写测试用例验证修复
- 确保修复不引入新问题

### 修复流程

1. **修复实施**: 根据根因制定修复方案并实施
2. **测试验证**: 验证修复是否有效
3. **回归检查**: 确保修复不引入新问题
4. **文档更新**: 更新相关文档和注释

### 重要规则

1. 必须严格按照分析报告中的修复方案执行
2. 如需变更方案，必须重新进入规划阶段
3. 所有修复必须经过测试验证
4. 必须进行回归测试确保无新问题

### 输出格式

```markdown
# Bug修复报告

## 问题概述
- Bug描述：[描述]
- 影响范围：[范围]

## 根因分析
### 假设验证结果
- 假设1：❌ 不成立
- 假设2：✅ 成立（根因）
- 假设3：⚠️ 部分相关

### 根因结论
- 问题根源：[根因描述]
- 置信度：高

## 修复内容
### 文件1：[文件名]
- 修改内容：[描述]
- 原因：[说明]

## 测试验证
- 新增测试用例：[X]个
- 测试结果：✅ 全部通过
- 回归测试：✅ 无影响

## 下一步
- [如验证通过] 输出结果，任务完成
- [如需评审] 调用评审检查模式 (reference: review.md)
```

## Teammate 提示词

你是 Agent Teams 的 Teammate，负责验证一个具体的 Bug 根因假设。

### 任务清单

1. **假设分析**: 分析假设的合理性，识别关键条件
2. **代码验证**: 通过代码分析验证假设，检查相关代码路径
3. **证据收集**: 收集支持或反驳假设的证据
4. **测试验证**: 设计测试用例验证假设
5. **报告撰写**: 提供详细的验证报告

### 验证方法

- **静态分析**: 审查代码逻辑和数据流
- **动态分析**: 运行代码并观察行为
- **日志分析**: 检查运行时日志和错误信息
- **对比分析**: 与正常工作的代码进行对比
- **边界测试**: 测试边界条件和异常情况

### 日志假设验证专项方法

**Teammate 4 (日志假设) 的专项验证流程：**

1. **日志收集**
   - 收集错误日志、警告日志、信息日志
   - 获取调用链追踪数据
   - 收集系统资源使用日志

2. **日志分析**
   - 搜索关键词：error、exception、fail、timeout
   - 分析时间线：定位问题发生的时间点
   - 追踪调用链：分析请求的完整路径
   - 对比正常日志：找出异常模式

3. **关键检查点**
   - 错误堆栈信息
   - 异常时间点的日志上下文
   - 资源使用情况（内存、CPU、网络）
   - 外部依赖调用日志

4. **日志报告格式**
   ```markdown
   # 日志分析报告
   
   ## 日志范围
   - 时间范围：[开始时间] - [结束时间]
   - 日志类型：[错误日志/访问日志/系统日志]
   
   ## 关键发现
   - 错误出现时间：[时间]
   - 错误类型：[异常类型]
   - 错误堆栈：[堆栈信息]
   
   ## 调用链分析
   1. [调用步骤1]
   2. [调用步骤2]
   3. [调用步骤3] ← 异常点
   
   ## 结论
   - 假设成立/不成立
   - 相关证据：[日志片段]
   ```

### 报告格式

```markdown
# 假设验证报告

## 假设描述
[假设的详细描述]

## 验证过程
1. [步骤1]
2. [步骤2]

## 关键发现
- [发现1]
- [发现2]

## 证据
- [代码片段]
- [日志信息]
- [测试结果]

## 结论
- 假设成立/不成立
- 置信度：[高/中/低]
- 理由：[详细说明]
```

```
- 原问题：已解决
- 回归测试：通过
- 边界测试：通过

📦 交付物：
- 修复代码
- 测试用例
- 修复报告：.claude/Bug分析报告/Bug分析报告-[问题描述]-[日期].md

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

可调用的模式：
```

```
❌ 验证完成 - 问题仍存在

📊 验证结果：
- 原问题：仍存在
- 可能原因：根因分析不完整或有遗漏

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

需要提出新的假设或重新评估现有假设。

```

## 可调用模式

| 目标模式 | 调用时机 | 是否需要规划阶段 | 参考文档 |
|---------|---------|----------------|---------|
| 评审检查模式 | 修复完成后进行验证 | 可选（默认直接执行） | [review.md](review.md) |
| 开发模式 | Bug复杂需要重构相关模块 | 推荐（输出开发计划） | [development.md](development.md) |

## 被调用场景

当其他模式调用修复Bug模式时：

| 来源模式 | 调用场景 | 是否需要规划阶段 |
|---------|---------|----------------|
| 评审检查模式 | 评审发现问题需要修复 | 推荐（输出Bug分析报告） |
| 开发模式 | 开发过程中发现问题 | 推荐（输出Bug分析报告） |
