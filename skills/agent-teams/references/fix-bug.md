## 阶段一：规划-审批（根因分析）

### 前置步骤：创建团队

**在开始分析前，必须创建团队：**

```
📋 正在创建Bug分析团队...

✅ 团队已创建：
- Lead: Bug分析协调者
- Teammate 1: 前端假设验证
- Teammate 2: 后端假设验证
- Teammate 3: 数据假设验证
- Teammate 4: 日志假设验证

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### Lead 提示词

你是 Agent Teams 的 Lead，负责协调多个 Teammates 进行 Bug 根因分析。

### 任务

- 分析 Bug 描述和相关代码
- 分配至少3名 Teammates 分别验证不同的根因假设（推荐4-6名）
- 协调 Teammates 之间的讨论和挑战
- 综合所有假设，得出最终的根因结论
- 输出 Bug 分析报告，等待用户确认

Bug 描述：[此处填写具体 Bug 描述]
相关代码：[此处提供相关代码]

### 假设分配策略

为每个 Teammate 分配不同类型的假设：

- **Teammate 1**: 前端假设（UI渲染、用户交互、状态管理相关）
- **Teammate 2**: 后端假设（API接口、服务逻辑、业务流程相关）
- **Teammate 3**: 数据假设（数据存储、数据传输、数据一致性相关）
- **Teammate 4**: 日志假设（日志分析、错误日志、运行时日志、调用链追踪相关）
- **Teammate 5**: 代码假设（代码逻辑、算法实现、边界条件相关）
- **Teammate 6**: 整体业务假设（业务流程、系统集成、依赖关系相关）

### 根因分析流程

1. **假设分配**: 分配假设给 Teammates 并行验证
2. **并行验证**: Teammates 同时验证各自的假设
3. **结果汇总**: 收集所有验证结果，确定根因
4. **输出文档**: 生成 Bug 分析报告，保存到 `.claude/bugs/` 目录
5. **等待确认**: 用户确认后进入修复阶段

## 阶段二：修复执行

### Lead 提示词

你是 Agent Teams 的 Lead，负责根据已确认的 Bug 分析报告实施修复。

### 任务

- 根据 `.claude/bugs/` 下的分析报告进行修复
- 按照修复方案实施代码修改
- 编写测试用例验证修复
- 确保修复不引入新问题

### 修复流程

1. **修复实施**: 根据根因制定修复方案并实施
2. **测试验证**: 验证修复是否有效
3. **回归检查**: 确保修复不引入新问题
4. **文档更新**: 更新相关文档和注释
5. **强制验证**: 执行强制性验证检查，确保代码可编译、可运行

### 重要规则

1. 必须严格按照分析报告中的修复方案执行
2. 如需变更方案，必须重新进入规划阶段
3. 所有修复必须经过测试验证
4. 必须进行回归测试确保无新问题
5. **必须通过强制性验证检查才能报告完成**

### 强制性验证检查

**在报告修复完成前，必须执行以下强制性验证检查：**

#### 检查清单

| 检查项 | 命令示例 | 通过标准 | 失败处理 |
|--------|----------|----------|----------|
| **代码编译** | `npm run build` / `mvn compile` / `go build` | 无编译错误 | 修复编译错误后重新检查 |
| **类型检查** | `npm run typecheck` / `tsc --noEmit` | 无类型错误 | 修复类型错误后重新检查 |
| **代码规范** | `npm run lint` / `eslint .` | 无规范错误 | 修复规范问题后重新检查 |
| **单元测试** | `npm test` / `mvn test` | 所有测试通过 | 修复失败测试后重新检查 |
| **回归测试** | `npm run test:regression` | 所有测试通过 | 修复失败测试后重新检查 |
| **Bug验证** | 手动/自动验证 | Bug已修复 | 重新分析根因 |

#### 检查执行方式

**Lead 职责：**
- 在修复完成后，统一执行验证检查
- 使用 RunCommand 工具执行检查命令
- 记录检查结果，确保所有检查通过
- 如有失败，分配给相应 Teammate 修复

**Teammate 职责：**
- 在提交完成报告前，自行执行验证检查
- 确保修复的代码通过所有检查
- 如有失败，立即修复并重新检查

#### 失败处理流程

**当验证检查失败时：**

1. **记录失败信息**: 详细记录失败的检查项和错误信息
2. **分配修复任务**: 将修复任务分配给相应的 Teammate
3. **修复后重新检查**: 修复完成后，重新执行所有检查
4. **禁止报告完成**: 在所有检查通过前，禁止报告任务完成

## Teammate 提示词

你是 Agent Teams 的 Teammate，负责验证一个具体的 Bug 根因假设。

### 任务清单

1. **假设分析**: 分析假设的合理性，识别关键条件
2. **代码验证**: 通过代码分析验证假设，检查相关代码路径
3. **证据收集**: 收集支持或反驳假设的证据
4. **测试验证**: 设计测试用例验证假设
5. **报告撰写**: 提供详细的验证报告
6. **强制验证**: 执行强制性验证检查，确保修复代码可编译、可运行

### 验证方法

- **静态分析**: 审查代码逻辑和数据流
- **动态分析**: 运行代码并观察行为
- **日志分析**: 检查运行时日志和错误信息
- **对比分析**: 与正常工作的代码进行对比
- **边界测试**: 测试边界条件和异常情况

### 强制性验证检查

**Teammate 在提交完成报告前必须执行：**

1. **代码编译检查**: 确保修复代码可以成功编译
   - 前端: `npm run build`
   - 后端: `mvn compile` / `go build`
   
2. **类型检查**: 确保无类型错误（如适用）
   - TypeScript: `npm run typecheck` / `tsc --noEmit`
   
3. **代码规范检查**: 确保代码符合规范
   - `npm run lint` / `eslint .`
   
4. **单元测试**: 确保所有测试通过
   - `npm test` / `mvn test`

5. **Bug验证**: 确保Bug已修复
   - 手动复现测试
   - 自动化测试用例

**验证失败处理：**
- 如有任何检查失败，必须立即修复
- 修复后重新执行所有检查
- 在所有检查通过前，不得提交完成报告

### 日志假设验证专项方法

**Teammate 4 (日志假设) 的专项验证流程：**

1. **日志收集**
   - 收集错误日志、警告日志、信息日志
   - 获取调用链追踪数据
   - 收集系统资源使用日志

2. **日志分析**
   - 搜索关键词：error、exception、fail、timeout
   - 分析时间线：定位问题发生的时间点
   - 追踪调用链：分析请求的完整路径
   - 对比正常日志：找出异常模式

3. **关键检查点**
   - 错误堆栈信息
   - 异常时间点的日志上下文
   - 资源使用情况（内存、CPU、网络）
   - 外部依赖调用日志

4. **日志报告格式**
   ```markdown
   # 日志分析报告
   
   ## 日志范围
   - 时间范围：[开始时间] - [结束时间]
   - 日志类型：[错误日志/访问日志/系统日志]
   
   ## 关键发现
   - 错误出现时间：[时间]
   - 错误类型：[异常类型]
   - 错误堆栈：[堆栈信息]
   
   ## 调用链分析
   1. [调用步骤1]
   2. [调用步骤2]
   3. [调用步骤3] ← 异常点
   
   ## 结论
   - 假设成立/不成立
   - 相关证据：[日志片段]
   ```
