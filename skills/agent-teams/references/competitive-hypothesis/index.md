# 竞争假设模式

适用于修复bug类任务，通过多个Teammates并行验证不同的根因假设，快速定位问题根源。

---

## 完整工作流程

```
┌──────────────────┐    确认根因    ┌──────────────────┐    验证通过    ┌──────────────────┐
│   阶段一         │ ────────────→ │   阶段二         │ ────────────→ │   阶段三         │
│   根因分析       │               │   修复实施       │               │   验证完成       │
│   (竞争假设)     │               │                  │               │                  │
└──────────────────┘               └──────────────────┘               └──────────────────┘
         │                                │                                │
         │ 输出:                          │ 输出:                          │ 输出:
         │ 根因结论                       │ 修复代码                        │ 验证结果
         │ 验证报告                       │ 测试用例                        │ 问题解决确认
         │                                │                                │
         └────────────────────────────────┴────────────────────────────────┘
                                      │
                                      ▼
                             ┌──────────────┐
                             │ 验证结果？   │
                             └──────────────┘
                               │         │
                         通过  │         │ 问题仍存在
                               ▼         ▼
                        ┌──────────┐  返回阶段一
                        │ 交付完成 │  (新假设)
                        └──────────┘
```

---

## 阶段文档

| 阶段 | 文档 | 说明 |
|------|------|------|
| 阶段一 | [analysis.md](analysis.md) | 根因分析 - 多假设并行验证 |
| 阶段二 | [fix.md](fix.md) | 修复实施 - 根据根因修复代码 |
| 阶段三 | [verify.md](verify.md) | 验证 - 确认问题已解决 |

---

## 阶段转换规则

| 当前阶段 | 完成标志 | 下一阶段 | 参考文档 |
|---------|---------|---------|---------|
| 根因分析 | 根因确认，置信度高 | 修复实施 | `fix.md` |
| 修复实施 | 代码修改完成，自测通过 | 验证 | `verify.md` |
| 验证 | 问题解决 | 交付完成 | - |
| 验证 | 问题仍存在 | 返回根因分析 | `analysis.md` |

---

## 快速开始

### Lead 触发流程

当用户选择"修复bug模式"时，Lead 应：

1. 询问 Bug 描述和相关代码
2. 进入**阶段一：根因分析**（reference: analysis.md）
3. 分配假设给 Teammates 并行验证

```
🐛 进入 Bug 修复模式

请提供：
1. Bug 描述
2. 相关代码/文件路径
3. 复现步骤（如有）

👉 开始根因分析 (reference: analysis.md)
```

---

## 完整流程示例

```
用户：修复一个登录失败的bug

Lead：开始根因分析，分配假设：
- Teammate 1: 前端假设（请求未正确发送）
- Teammate 2: 后端假设（认证逻辑问题）
- Teammate 3: 数据假设（用户数据问题）

Teammates 并行验证...

Lead：汇总结果
- 假设1：不成立，请求正常发送
- 假设2：✅ 成立，密码验证逻辑有bug
- 假设3：不成立，用户数据正常

根因确认：密码验证逻辑bug

→ 进入修复实施 (fix.md)

Teammate 2 实施修复...

Lead：修复完成，开始验证

→ 进入验证 (verify.md)

验证通过

✅ Bug 修复完成
```
