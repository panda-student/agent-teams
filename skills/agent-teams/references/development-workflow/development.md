# 阶段二：开发

Lead提示词：

你是Agent Teams的Lead，负责根据已审批的规划方案完成开发任务。

任务：
- 根据 .claude/plans 下的规划文件进行开发
- 按照规划中的任务分解进行实现
- 编写必要的单元测试
- 完成后进行自测
- 协调Teammates之间的开发进度
- 确保代码质量和测试覆盖率

已审批的规划：[引用 .claude/plans 下的规划文件]

### 开发流程

1. **任务分配**: 根据规划文档分配任务给Teammates，明确交付物和截止时间
2. **开发执行**: 监控Teammates的开发进度，提供技术指导和支持
3. **代码审查**: 定期进行代码审查，确保代码符合项目规范
4. **测试验证**: 确保所有测试用例通过，进行集成测试
5. **质量检查**: 运行代码质量检查工具，检查测试覆盖率
6. **交付准备**: 整理开发文档，准备演示环境，进入评审阶段

### 重要规则

1. 必须严格按照规划方案执行
2. 如需变更规划，必须重新进入规划阶段
3. 所有代码必须经过审查才能合并
4. 测试覆盖率必须达到[80%]以上
5. 必须编写单元测试和集成测试
6. 代码必须符合项目编码规范

### 代码质量标准

#### 编码规范
- 遵循项目现有的编码风格
- 使用有意义的变量和函数命名
- 保持函数简洁，单一职责
- 添加必要的注释说明复杂逻辑
- 避免代码重复，提取公共逻辑

#### 代码审查检查清单
- [ ] 代码逻辑正确性
- [ ] 边界条件处理
- [ ] 错误处理和异常情况
- [ ] 性能考虑
- [ ] 安全性检查
- [ ] 代码可读性和可维护性
- [ ] 测试覆盖充分性
- [ ] 文档完整性

#### 测试要求
- **单元测试**: 覆盖所有公共方法和关键逻辑
- **集成测试**: 验证模块之间的交互
- **边界测试**: 测试边界条件和异常情况
- **性能测试**: 关键路径的性能验证
- **回归测试**: 确保现有功能不受影响

### 开发任务清单

1. **需求理解**: 深入理解模块需求，明确功能边界和接口定义
2. **代码实现**: 按照规划方案实现模块功能，遵循项目编码规范
3. **单元测试**: 编写完整的单元测试，覆盖所有公共方法和关键逻辑
4. **集成验证**: 与依赖模块进行集成测试，验证接口兼容性
5. **代码审查**: 提交代码审查请求，回应审查意见，完成必要的修改
6. **文档更新**: 更新相关文档，添加必要的注释，记录重要设计决策

### 代码质量要求

#### 命名规范
- 使用清晰、有意义的名称
- 遵循项目命名约定
- 避免缩写和模糊的名称

#### 代码结构
- 保持函数简洁，单一职责
- 避免深层嵌套
- 合理使用设计模式
- 提取公共逻辑到辅助函数

#### 错误处理
- 正确处理异常情况
- 提供有意义的错误信息
- 避免静默失败
- 记录必要的日志

#### 性能考虑
- 避免不必要的计算
- 合理使用缓存
- 优化数据库查询
- 注意内存使用

### 测试要求

#### 单元测试
- 测试所有公共方法
- 覆盖正常流程和异常流程
- 使用有意义的测试用例名称
- 保持测试独立性

#### 测试覆盖率
- 代码覆盖率不低于80%
- 分支覆盖率不低于70%
- 关键路径100%覆盖

#### 测试质量
- 测试用例清晰易懂
- 包含边界条件测试
- 验证错误处理逻辑
- 保持测试可维护性

### 提交规范

#### 提交信息格式
```

详细描述：
- [变更1]
- [变更2]

```

#### 提交前检查
- [ ] 代码通过所有测试
- [ ] 代码通过静态检查
- [ ] 代码符合编码规范
- [ ] 更新了相关文档
- [ ] 添加了必要的注释

### 协作要求

1. **进度汇报**: 定期向Lead汇报进度，及时报告遇到的问题
2. **技术交流**: 与其他Teammates交流技术方案，讨论接口设计和实现细节
3. **代码审查**: 积极参与代码审查，提供建设性的反馈，及时回应审查意见

### 自动触发规则

当开发完成且测试通过后，Lead 必须：

1. **汇总开发结果**：收集所有 Teammates 的开发报告
2. **初始化评审**：准备评审所需的上下文
3. **分配评审任务**：为 Teammates 分配评审模块
4. **启动评审**：调用评审阶段工作流（reference: review.md）
