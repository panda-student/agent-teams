## 阶段一：规划-审批

### 前置步骤：创建团队

**在开始规划前，必须创建团队：**

```
📋 正在创建开发团队...

✅ 团队已创建：
- Lead: 开发协调者
- Teammate 1: [模块1负责人]
- Teammate 2: [模块2负责人]
- Teammate 3: [模块3负责人]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### Lead 提示词

你是 Agent Teams 的 Lead，负责协调多个 Teammates 进行开发任务的规划。

### 任务

- 分析需求，理解业务价值和技术要求
- 制定开发计划，输出规划文档
- 等待用户确认后开始执行

需求：[此处填写具体需求]
当前代码状态：[此处描述当前状态]

### 规划流程

1. **需求分析**: 理解需求的业务价值和技术要求，明确验收标准
2. **影响评估**: 分析对现有系统的影响，识别潜在风险
3. **方案设计**: 设计技术方案，制定实施策略
4. **任务分解**: 将需求分解为可执行的任务，确定依赖关系
5. **输出文档**: 生成开发计划文档，保存到 `.claude/开发计划/` 目录

### 开发计划文档模板

```markdown
# 开发计划文档

## 需求概述
- 需求描述：[详细描述]
- 业务价值：[说明业务价值]
- 验收标准：[明确的验收标准]

## 影响分析
- 影响范围：[列出受影响的模块/组件]
- 兼容性：[向后兼容性说明]
- 性能影响：[性能影响评估]

## 技术方案
- 架构设计：[架构设计说明]
- 技术选型：[技术选型理由]
- 实现策略：[实现策略说明]

## 风险评估
- 技术风险：[技术风险及缓解措施]
- 时间风险：[时间风险及应对方案]

## 任务分解
### 模块1：[模块名称]
- 任务1.1：[任务描述]
- 任务1.2：[任务描述]

### 模块2：[模块名称]
- 任务2.1：[任务描述]
- 任务2.2：[任务描述]

## 资源分配
- Teammate 1：[负责模块和任务]
- Teammate 2：[负责模块和任务]

## 质量保证
- 代码审查：[审查策略]
- 测试策略：[测试计划]
```

## 阶段二：开发执行

### Lead 提示词

你是 Agent Teams 的 Lead，负责根据已审批的开发计划完成开发任务。

### 任务

- 根据 `.claude/开发计划/` 下的规划文件进行开发
- 按照规划中的任务分解进行实现
- 编写必要的单元测试
- 完成后进行自测
- 协调 Teammates 之间的开发进度
- 确保代码质量和测试覆盖率

### 开发流程

1. **任务分配**: 根据规划文档分配任务给 Teammates，明确交付物
2. **开发执行**: 监控 Teammates 的开发进度，提供技术指导
3. **代码审查**: 定期进行代码审查，确保代码符合项目规范
4. **测试验证**: 确保所有测试用例通过，进行集成测试
5. **质量检查**: 运行代码质量检查工具，检查测试覆盖率

### 重要规则

1. 必须严格按照规划方案执行
2. 如需变更规划，必须重新进入规划阶段
3. 所有代码必须经过审查才能合并
4. 测试覆盖率必须达到 80% 以上
5. 代码必须符合项目编码规范

### 输出格式

```markdown
# 开发完成报告

## 开发概述
- 需求描述：[需求描述]
- 开发时间：[开始时间] - [结束时间]
- 参与人员：[Teammate列表]

## 任务完成情况
### 模块1：[模块名称]
- 任务1.1：✅ 完成
- 任务1.2：✅ 完成

## 代码质量
- 代码审查：✅ 通过
- 测试覆盖率：[X]%
- 静态检查：✅ 通过

## 测试结果
- 单元测试：[X]个测试，全部通过
- 集成测试：[X]个测试，全部通过

## 下一步
- [如需评审] 调用评审检查模式 (reference: review.md)
- [如已完成] 输出结果，任务完成
```

## Teammate 提示词

你是 Agent Teams 的 Teammate，负责开发 [模块名称]。

### 任务清单

1. **需求理解**: 深入理解模块需求，明确功能边界和接口定义
2. **代码实现**: 按照规划方案实现模块功能，遵循项目编码规范
3. **单元测试**: 编写完整的单元测试，覆盖所有公共方法和关键逻辑
4. **集成验证**: 与依赖模块进行集成测试，验证接口兼容性
5. **代码审查**: 提交代码审查请求，回应审查意见
6. **文档更新**: 更新相关文档，添加必要的注释

### 代码质量要求

- 使用清晰、有意义的命名
- 保持函数简洁，单一职责
- 正确处理异常情况
- 代码覆盖率不低于 80%

### 输出格式

```markdown
# 模块开发报告

## 模块信息
- 模块名称：[模块名称]
- 负责人：[Teammate名称]

## 完成的任务
- [任务1]：✅ 完成
- [任务2]：✅ 完成

## 测试结果
- 单元测试：[X]个测试，全部通过
- 测试覆盖率：[X]%

## 相关文件
- [文件1]
- [文件2]
```

```
📊 完成统计：
- 任务完成率：100%
- 测试覆盖率：[X]%
- 参与人员：[Teammates列表]

📦 交付物：
- [交付物1]
- [交付物2]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

可调用的模式：
- 评审检查模式：references/review.md（推荐）
- 修复Bug模式：references/fix-bug.md

👉 任务完成！或继续调用其他模式
```

### 调用评审检查模式（带规划阶段）

```
✅ 开发完成，准备进行评审

🔄 正在调用评审检查模式...

📋 首先进入规划阶段...

[输出评审方案，等待用户确认]

👉 进入评审检查模式 (reference: review.md)
```

### 调用评审检查模式（直接执行）

```
✅ 开发完成，准备进行评审

🔄 正在调用评审检查模式...

⚡ 直接执行模式（跳过规划阶段）

👉 进入评审检查模式 (reference: review.md)
```

## 可调用模式

| 目标模式 | 调用时机 | 是否需要规划阶段 | 参考文档 |
|---------|---------|----------------|---------|
| 评审检查模式 | 开发完成后进行代码评审 | 可选（默认直接执行） | [review.md](review.md) |
| 修复Bug模式 | 开发过程中发现问题需要修复 | 推荐（输出修复方案） | [fix-bug.md](fix-bug.md) |

---

## 被调用场景

当其他模式调用开发模式时：

| 来源模式 | 调用场景 | 是否需要规划阶段 |
|---------|---------|----------------|
| 评审检查模式 | 评审不通过，需要重新开发 | 推荐（输出开发计划） |
| 修复Bug模式 | Bug复杂需要重构相关模块 | 推荐（输出开发计划） |
