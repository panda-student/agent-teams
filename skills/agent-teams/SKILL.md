# Agent Teams 技能

## 触发行为

**当此技能被调用时，必须首先询问用户选择使用哪种模式：**

1. **开发需求模式** - 适用于新功能开发、需求实现等场景
2. **修复bug模式** - 适用于问题排查、bug修复等场景

根据用户选择的模式，进入相应的工作流程。

## 阶段转换规则

### 开发需求模式

| 当前阶段 | 完成标志 | 用户确认 | 下一阶段 | 参考文档 |
|---------|---------|---------|---------|---------|
| 规划 | 规划文档输出到 `.claude/plans/` | 需要 | 开发 | `references/development-workflow/development.md` |
| 开发 | 代码实现完成，测试通过 | 不需要 | 评审 | `references/development-workflow/review.md` |
| 评审 | 评审报告输出 | - | 交付或返工 | - |

### 修复Bug模式

| 当前阶段 | 完成标志 | 下一阶段 |
|---------|---------|---------|
| 根因分析 | 根因确认，置信度高 | 修复实施 |
| 修复实施 | 代码修复完成 | 验证 |
| 验证 | 问题解决 | 交付完成 |

## 核心组件

- **Lead**: 协调者，负责任务分配、进度跟踪和结果整合
- **Teammates**: 执行者，负责具体任务的实施和交付
- **Task List**: 共享任务列表，用于跟踪团队进度
- **Mailbox**: 消息系统，用于团队成员之间的沟通

## 使用场景

1. **开发需求** → 开发工作流（详见 [development-workflow/planning.md](references/development-workflow/planning.md)）
2. **修复bug** → 竞争假设模式（详见 [competitive-hypothesis/index.md](references/competitive-hypothesis/index.md)）

## 阶段转换提示词

### 规划 → 开发

当规划完成且用户确认后，Lead 必须输出：

```
📋 规划已完成，文档保存在 .claude/plans/

正在启动开发阶段...
- 读取规划文档
- 分配开发任务给 Teammates
- 初始化任务列表

👉 进入开发阶段
```

### 开发 → 评审

当开发完成后，Lead 必须输出：

```
✅ 开发已完成

正在进行代码评审...
- 分配评审任务
- 检查代码质量
- 验证测试覆盖

👉 进入评审阶段 (reference: references/development-workflow/review.md)
```

### 评审 → 交付/返工

评审完成后，Lead 必须输出：

```
📝 评审报告：
- 通过项：[列表]
- 问题项：[列表]
- 总体结论：[通过/需修改]

[如果通过] 👉 准备交付
[如果需修改] 👉 返回开发阶段修复问题
```

## 最佳实践

1. **上下文管理**: 为Teammates提供完整的上下文信息
2. **任务拆分**: 每个Teammate负责5-6个任务，明确交付物
3. **避免冲突**: 每个Teammate拥有不同的文件集
4. **监控与引导**: Lead定期检查进度，及时解决问题
5. **协作机制**: Lead等待Teammates完成后再继续下一步
6. **质量保证**: 每个阶段完成后进行评审，确保代码质量
7. **阶段转换**: 明确告知用户当前阶段和下一阶段，确保流程连贯
