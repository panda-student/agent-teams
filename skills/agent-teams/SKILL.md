## 触发行为

**当此技能被调用时，Lead 必须执行以下步骤：**

### 步骤一：创建团队（强制）

**无论何种模式，必须首先创建团队：**

```
📋 正在创建团队...

✅ 团队已创建：
- Lead: [协调者]
- Teammate 1: [角色]
- Teammate 2: [角色]
- Teammate 3: [角色]
- ... (至少3名成员)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**团队规模要求：**
- 最少3名成员（不含Lead）
- 推荐3-5名成员
- 根据任务复杂度可适当增加

### 步骤二：根据需求明确程度采取策略

### 策略一：需求明确 → 自动判断直接进入

当用户需求明确时，Lead 自动判断模式并直接进入：

**判断规则：**

| 关键词/场景 | 自动选择模式 |
|------------|-------------|
| 开发、实现、新增功能、重构、添加 | 开发模式 |
| bug、修复、问题、错误、缺陷、异常 | 修复Bug模式 |
| 评审、检查、审查、审计、代码质量 | 评审检查模式 |

**执行流程：**

```
📋 正在分析需求...

需求类型：[开发/修复Bug/评审检查]

✅ 自动选择：[模式名称]

🚀 正在进入规划阶段...

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

👉 进入 [模式名称] (reference: references/[文件名].md)
```

**示例：**

| 用户输入 | 自动选择 |
|---------|---------|
| "帮我开发一个登录功能" | 开发模式 |
| "有个bug需要修复" | 修复Bug模式 |
| "帮我评审这段代码" | 评审检查模式 |
| "实现用户管理模块" | 开发模式 |
| "修复登录失败的问题" | 修复Bug模式 |

### 策略二：需求不明确 → 让用户选择

当用户需求不明确或无法判断时，使用 AskUserQuestion 工具让用户选择：

**触发条件：**
- 用户只说"帮我处理一下"等模糊表述
- 用户需求涉及多种模式
- 无法从上下文判断意图

**执行流程：**

```
AskUserQuestion:
  question: "请选择执行模式："
  header: "模式选择"
  options:
    - label: "📦 开发模式"
      description: "新功能开发、需求实现、代码重构"
    - label: "🐛 修复Bug模式"
      description: "问题排查、bug修复、缺陷处理"
    - label: "🔍 评审检查模式"
      description: "代码评审、质量检查、安全审计"
```

用户选择后：

```
✅ 您选择了：[模式名称]

🚀 正在进入规划阶段...

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

👉 进入 [模式名称] (reference: references/[文件名].md)
```

## 核心设计原则

### 1. 规划-审批阶段（通用）

每个执行模式都包含规划-审批阶段，确保用户对执行计划有充分了解和确认：

```
┌─────────────────────────────────────────────────────────────┐
│                    规划-审批阶段                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌──────────┐    输出文档    ┌──────────┐    用户确认       │
│  │ 分析需求  │ ───────────→ │ 等待确认  │ ───────────→      │
│  │          │              │          │                   │
│  └──────────┘              └──────────┘                   │
│       │                                                    │
│       │ 输出:                                              │
│       │ - 计划文档                                         │
│       │ - 预期产出                                         │
│       │ - 风险评估                                         │
│       │                                                    │
│       └────────────────────────────────────────────────────┘
│                           │                                 │
│                           ▼                                 │
│                  用户确认后执行                              │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 2. 文档输出位置

| 模式 | 文档类型 | 输出位置 | 文档命名 |
|------|----------|----------|----------|
| 开发模式 | 开发计划 | `.claude/plans/` | `开发计划-[需求名称]-[日期].md` |
| 修复Bug模式 | Bug分析报告 | `.claude/bugs/` | `Bug分析报告-[问题描述]-[日期].md` |
| 评审检查模式 | 评审方案 | `.claude/reviews/` | `评审方案-[评审范围]-[日期].md` |

**命名示例：**

| 模式 | 文档名称示例 |
|------|-------------|
| 开发模式 | `开发计划-用户登录功能-2024-01-15.md` |
| 修复Bug模式 | `Bug分析报告-头像上传失败-2024-01-15.md` |
| 评审检查模式 | `评审方案-支付模块代码-2024-01-15.md` |

### 3. 报告文件复用机制

**当处理同一批问题时，应复用已有的报告文件，而不是创建新文件：**

#### 复用规则

| 场景 | 处理方式 | 说明 |
|------|----------|------|
| 同一问题的后续修复 | 复用原报告文件 | 在原文件中追加修复记录 |
| 同一批Bug的多次修复 | 复用原报告文件 | 更新修复状态，追加新的修复内容 |
| 评审发现问题后修复 | 复用原评审报告 | 在评审报告中追加修复记录 |
| 同一需求的迭代开发 | 复用原开发计划 | 更新计划内容，追加迭代记录 |

#### 文件复用判断标准

**判断是否为"同一批问题"的标准：**

1. **问题标识相同**: 问题/需求的核心标识符相同
2. **时间窗口**: 在合理的时间窗口内（建议7天内）
3. **上下文关联**: 属于同一任务链或工作流
4. **用户确认**: 用户明确表示是同一问题的延续

#### 实现方式

```
┌─────────────────────────────────────────────────────────────┐
│                    报告文件复用流程                           │
├─────────────────────────────────────────────────────────────┤
│                                                              │

│  │ 检查是否存在  │                                           │

│         ▼                                                    │
│  ┌──────────────┐    是     ┌──────────────┐               │

│  │ 问题？      │           │ 追加内容     │               │
│  └──────┬───────┘           └──────────────┘               │
```

### 4. 模式间调用规则

**任意模式都可以调用其他模式，调用时可以选择：**

- **带规划阶段**：先输出计划文档，等待用户确认后再执行
- **直接执行**：跳过规划阶段，直接进入执行（适用于紧急情况或用户明确指示）

| 调用方式 | 说明 | 使用场景 |

#### 调用示例

**示例1: Teammate 在开发任务中调用 commit 技能**

```
Teammate 1: 负责用户登录模块开发

任务执行:
1. 实现登录功能代码
2. 编写单元测试
3. 调用 commit 技能进行代码提交
   - Skill: commit
   - 自动分析变更并分类提交
4. 发送完成报告给 Lead
```

**示例2: Teammate 在Bug修复中调用 pgsql-operator 技能**

```
Teammate 2: 负责数据库相关Bug修复

任务执行:
1. 分析数据库问题
2. 调用 pgsql-operator 技能
   - Skill: pgsql-operator
   - 执行数据库查询和修复
3. 验证修复结果
4. 发送完成报告给 Lead
```

#### 调用规则

1. **明确需求**: Teammate 需要明确任务是否需要特定技能支持
2. **技能匹配**: 选择合适的技能进行调用
3. **结果验证**: 调用技能后需要验证执行结果
4. **报告整合**: 将技能执行结果整合到任务报告中
5. **错误处理**: 技能调用失败时需要回退到手动执行

#### Lead 协调职责

**Lead 在技能调用中的职责：**

- **任务分配时**: 明确指出需要调用的技能
- **资源协调**: 避免多个 Teammate 同时调用同一技能造成冲突
- **结果审核**: 审核技能执行结果是否符合要求
- **问题处理**: 处理技能调用失败的情况

## 质量保证机制

### 强制性验证检查标准

**所有模式在报告任务完成前，都必须执行强制性验证检查：**

#### 通用检查项

| 检查项 | 优先级 | 命令示例 | 通过标准 |
|--------|--------|----------|----------|
| **代码编译** | P0 | `npm run build` / `mvn compile` | 无编译错误 |
| **类型检查** | P0 | `npm run typecheck` / `tsc --noEmit` | 无类型错误 |
| **代码规范** | P1 | `npm run lint` / `eslint .` | 无规范错误 |
| **单元测试** | P0 | `npm test` / `mvn test` | 所有测试通过 |

#### 模式特定检查项

| 模式 | 额外检查项 | 通过标准 |
|------|-----------|----------|
| **开发模式** | 集成测试 | 所有测试通过 |
| **修复Bug模式** | 回归测试 + Bug验证 | 所有测试通过 + Bug已修复 |
| **评审检查模式** | 评审报告完整性 | 包含所有评审维度 |

#### 验证检查执行原则

1. **强制性**: 所有检查项必须通过，缺一不可
2. **顺序性**: 按优先级顺序执行，P0优先
3. **完整性**: 不能跳过任何检查项
4. **可追溯**: 记录所有检查结果
5. **失败处理**: 任何检查失败必须修复后重新执行所有检查

## 任务状态同步机制

### 状态定义

| 状态 | 说明 |
|------|------|
| `pending` | 任务待执行 |
| `in_progress` | 任务执行中 |
| `completed` | 任务已完成 |
| `blocked` | 任务阻塞 |

### Teammate 完成报告标准

**每个 Teammate 完成任务后必须发送完成报告：**

```
📤 [Teammate名称] 任务完成报告

任务ID: [任务ID]
状态: completed
完成时间: [时间戳]

交付物:
- [交付物1]
- [交付物2]

相关文件:
- [文件路径1]
- [文件路径2]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### Lead 状态检测流程

```
┌─────────────────────────────────────────────────────────────┐
│                    任务状态同步流程                           │
├─────────────────────────────────────────────────────────────┤
│                                                              │

│  │ 分配任务     │                                           │
│  └──────┬───────┘                                           │
│         │                                                    │
│         ▼                                                    │
│  ┌──────────────┐    收到报告    ┌──────────────┐          │
│  │ 等待完成报告 │ ────────────→ │ 更新任务状态  │          │
│  └──────┬───────┘              └──────┬───────┘          │
│         │                             │                    │

```

### 防卡住机制

**当检测到任务状态不一致时：**

1. **主动查询**: Lead 定期查询 Teammate 的执行状态
2. **状态同步**: 如果 Teammate 已完成但未报告，主动更新状态
3. **超时处理**: 设置超时时间，超时后检查实际执行情况
4. **异常恢复**: 发现卡住时，重新分配或提醒执行

**查询命令格式：**

```
🔍 Lead 查询任务状态

任务ID: [任务ID]
当前状态: [状态]
请 Teammate [名称] 确认执行情况
```

## 最佳实践

1. **上下文管理**: 为Teammates提供完整的上下文信息
2. **任务拆分**: 每个Teammate负责5-6个任务，明确交付物
3. **避免冲突**: 每个Teammate拥有不同的文件集
4. **监控与引导**: Lead定期检查进度，及时解决问题
5. **协作机制**: Lead等待Teammates完成后再继续下一步
6. **质量保证**: 每个阶段完成后进行评审，确保代码质量
7. **模式切换**: 明确告知用户当前模式和切换原因
8. **文档留存**: 所有计划文档保存到对应目录，便于追溯
