## 触发行为

**当此技能被调用时，Lead 必须执行以下步骤：**

### 步骤一：创建团队（强制）

**无论何种模式，必须首先创建团队：**

```
📋 正在创建团队...

✅ 团队已创建：
- Lead: [协调者]
- Teammate 1: [角色]
- Teammate 2: [角色]
- Teammate 3: [角色]
- ... (至少3名成员)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**团队规模要求：**
- 最少3名成员（不含Lead）
- 推荐3-5名成员
- 根据任务复杂度可适当增加

### 步骤二：根据需求明确程度采取策略

### 策略一：需求明确 → 自动判断直接进入

当用户需求明确时，Lead 自动判断模式并直接进入：

**判断规则：**

| 关键词/场景 | 自动选择模式 |
|------------|-------------|
| 开发、实现、新增功能、重构、添加 | 开发模式 |
| bug、修复、问题、错误、缺陷、异常 | 修复Bug模式 |
| 评审、检查、审查、审计、代码质量 | 评审检查模式 |

**执行流程：**

```
📋 正在分析需求...

需求类型：[开发/修复Bug/评审检查]

✅ 自动选择：[模式名称]

🚀 正在进入规划阶段...

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

👉 进入 [模式名称] (reference: references/[文件名].md)
```

**示例：**

| 用户输入 | 自动选择 |
|---------|---------|
| "帮我开发一个登录功能" | 开发模式 |
| "有个bug需要修复" | 修复Bug模式 |
| "帮我评审这段代码" | 评审检查模式 |
| "实现用户管理模块" | 开发模式 |
| "修复登录失败的问题" | 修复Bug模式 |

### 策略二：需求不明确 → 让用户选择

当用户需求不明确或无法判断时，使用 AskUserQuestion 工具让用户选择：

**触发条件：**
- 用户只说"帮我处理一下"等模糊表述
- 用户需求涉及多种模式
- 无法从上下文判断意图

**执行流程：**

```
AskUserQuestion:
  question: "请选择执行模式："
  header: "模式选择"
  options:
    - label: "📦 开发模式"
      description: "新功能开发、需求实现、代码重构"
    - label: "🐛 修复Bug模式"
      description: "问题排查、bug修复、缺陷处理"
    - label: "🔍 评审检查模式"
      description: "代码评审、质量检查、安全审计"
```

用户选择后：

```
✅ 您选择了：[模式名称]

🚀 正在进入规划阶段...

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

👉 进入 [模式名称] (reference: references/[文件名].md)
```

## 核心设计原则

### 1. 规划-审批阶段（通用）

每个执行模式都包含规划-审批阶段，确保用户对执行计划有充分了解和确认：

```
┌─────────────────────────────────────────────────────────────┐
│                    规划-审批阶段                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌──────────┐    输出文档    ┌──────────┐    用户确认       │
│  │ 分析需求  │ ───────────→ │ 等待确认  │ ───────────→      │
│  │          │              │          │                   │
│  └──────────┘              └──────────┘                   │
│       │                                                    │
│       │ 输出:                                              │
│       │ - 计划文档                                         │
│       │ - 预期产出                                         │
│       │ - 风险评估                                         │
│       │                                                    │
│       └────────────────────────────────────────────────────┘
│                           │                                 │
│                           ▼                                 │
│                  用户确认后执行                              │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 2. 文档输出位置

| 模式 | 文档类型 | 输出位置 | 文档命名 |
|------|----------|----------|----------|
| 开发模式 | 开发计划 | `.claude/开发计划/` | `开发计划-[需求名称]-[日期].md` |
| 修复Bug模式 | Bug分析报告 | `.claude/Bug分析报告/` | `Bug分析报告-[问题描述]-[日期].md` |
| 评审检查模式 | 评审方案 | `.claude/评审方案/` | `评审方案-[评审范围]-[日期].md` |

**命名示例：**

| 模式 | 文档名称示例 |
|------|-------------|
| 开发模式 | `开发计划-用户登录功能-2024-01-15.md` |
| 修复Bug模式 | `Bug分析报告-头像上传失败-2024-01-15.md` |
| 评审检查模式 | `评审方案-支付模块代码-2024-01-15.md` |

### 3. 模式间调用规则

**任意模式都可以调用其他模式，调用时可以选择：**

- **带规划阶段**：先输出计划文档，等待用户确认后再执行
- **直接执行**：跳过规划阶段，直接进入执行（适用于紧急情况或用户明确指示）

### 调用场景详解

| 当前模式 | 目标模式 | 调用时机 | 是否需要规划阶段 |
|---------|---------|---------|----------------|
| 开发模式 | 评审检查模式 | 开发完成后 | 可选（默认直接执行） |
| 评审检查模式 | 修复Bug模式 | 发现问题需要修复 | 推荐（输出修复方案） |
| 评审检查模式 | 开发模式 | 需要重新开发或重构 | 推荐（输出开发计划） |
| 修复Bug模式 | 评审检查模式 | 修复完成后验证 | 可选（默认直接执行） |
| 修复Bug模式 | 开发模式 | 问题复杂需要重构 | 推荐（输出开发计划） |

## 核心组件

- **Lead**: 协调者，负责任务分配、进度跟踪和结果整合
- **Teammates**: 执行者，负责具体任务的实施和交付
- **Task List**: 共享任务列表，用于跟踪团队进度
- **Mailbox**: 消息系统，用于团队成员之间的沟通

## 任务状态同步机制

### 状态定义

| 状态 | 说明 |
|------|------|
| `pending` | 任务待执行 |
| `in_progress` | 任务执行中 |
| `completed` | 任务已完成 |
| `blocked` | 任务阻塞 |

### Teammate 完成报告标准

**每个 Teammate 完成任务后必须发送完成报告：**

```
📤 [Teammate名称] 任务完成报告

任务ID: [任务ID]
状态: completed
完成时间: [时间戳]

交付物:
- [交付物1]
- [交付物2]

相关文件:
- [文件路径1]
- [文件路径2]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### Lead 状态检测流程

```
┌─────────────────────────────────────────────────────────────┐
│                    任务状态同步流程                           │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌──────────────┐                                           │
│  │ 分配任务     │                                           │
│  └──────┬───────┘                                           │
│         │                                                    │
│         ▼                                                    │
│  ┌──────────────┐    收到报告    ┌──────────────┐          │
│  │ 等待完成报告 │ ────────────→ │ 更新任务状态  │          │
│  └──────┬───────┘              └──────┬───────┘          │
│         │                             │                    │
│         │ 超时检测                    │                    │
│         ▼                             ▼                    │
│  ┌──────────────┐              ┌──────────────┐          │
│  │ 检查Teammate │              │ 检查是否全部  │          │
│  │ 是否在执行   │              │ 完成         │          │
│  └──────┬───────┘              └──────┬───────┘          │
│         │                             │                    │
│         │ 未执行                      │ 是                 │
│         ▼                             ▼                    │
│  ┌──────────────┐              ┌──────────────┐          │
│  │ 重新分配或   │              │ 汇总结果     │          │
│  │ 提醒执行     │              │              │          │
│  └──────────────┘              └──────────────┘          │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 防卡住机制

**当检测到任务状态不一致时：**

1. **主动查询**: Lead 定期查询 Teammate 的执行状态
2. **状态同步**: 如果 Teammate 已完成但未报告，主动更新状态
3. **超时处理**: 设置超时时间，超时后检查实际执行情况
4. **异常恢复**: 发现卡住时，重新分配或提醒执行

**查询命令格式：**

```
🔍 Lead 查询任务状态

任务ID: [任务ID]
当前状态: [状态]
请 Teammate [名称] 确认执行情况
```

## 通用工作流程

### 阶段一：规划-审批

Lead 分析需求，输出计划文档：

```
📋 正在分析需求...

需求类型：[开发/修复Bug/评审检查]

📄 计划文档已生成：[文档路径]

执行计划：
1. [步骤1]
2. [步骤2]
3. [步骤3]

预期产出：
- [产出物1]
- [产出物2]

风险评估：
- [风险1]：[缓解措施]
- [风险2]：[缓解措施]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

请确认是否按此计划执行？
- 确认：输入"确认"或"开始"
- 修改：说明需要调整的内容
- 切换模式：指定其他模式（开发/修复Bug/评审检查）
```

### 阶段二：执行

用户确认后，Lead 开始执行：

```
✅ 计划已确认

🚀 开始执行 [模式名称]...

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[执行过程...]

👉 执行中...
```

### 阶段三：输出与转换

执行完成后，输出结果并提示可调用的模式：

```
🎉 [模式名称] 执行完成

📊 执行结果：
- [结果1]
- [结果2]

📦 交付物：
- [交付物1]
- [交付物2]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

可调用的模式：
- 开发模式：references/development.md
- 修复Bug模式：references/fix-bug.md
- 评审检查模式：references/review.md

👉 任务完成！或继续调用其他模式
```

## 最佳实践

1. **上下文管理**: 为Teammates提供完整的上下文信息
2. **任务拆分**: 每个Teammate负责5-6个任务，明确交付物
3. **避免冲突**: 每个Teammate拥有不同的文件集
4. **监控与引导**: Lead定期检查进度，及时解决问题
5. **协作机制**: Lead等待Teammates完成后再继续下一步
6. **质量保证**: 每个阶段完成后进行评审，确保代码质量
7. **模式切换**: 明确告知用户当前模式和切换原因
8. **文档留存**: 所有计划文档保存到对应目录，便于追溯
