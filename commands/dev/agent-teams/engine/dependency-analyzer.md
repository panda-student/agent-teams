# 依赖分析器 (Dependency Analyzer)

依赖分析器是并行执行引擎的第二个核心组件，负责分析任务间的依赖关系，构建依赖图，并划分并行执行组。

---

## 分析算法

### 算法入口

```
输入: 任务分解表 (Task Decomposition Table)
输出: 依赖图 (DAG) + 并行执行组 (Parallel Execution Groups)
```

### 分析步骤

```
步骤1: 依赖类型识别
    │
    ├── 硬依赖 (Hard Dependency)
    │   - 任务B必须在任务A完成后才能开始
    │   - 原因: 输出依赖、资源依赖、顺序约束
    │   - 标记: A → B (实线)
    │
    ├── 软依赖 (Soft Dependency)
    │   - 任务B建议在任务A后执行，但可并行
    │   - 原因: 效率优化、资源竞争
    │   - 标记: A ⇢ B (虚线)
    │
    └── 无依赖 (No Dependency)
        - 任务A和B完全独立
        - 可并行执行
        - 标记: A, B (同组)

步骤2: 依赖图构建 (DAG)
    │
    ├── 节点创建
    │   - 每个任务是一个节点
    │   - 节点属性: ID, 名称, 文件范围, 复杂度
    │
    ├── 边创建
    │   - 根据依赖类型创建边
    │   - 边属性: 类型(硬/软), 原因
    │
    └── 环检测
        - 确保无环 (DAG性质)
        - 有环则报错并提示

步骤3: 关键路径计算
    │
    ├── 正向遍历
    │   - 计算每个任务的最早开始时间 (ES)
    │   - 计算每个任务的最早结束时间 (EF)
    │
    ├── 反向遍历
    │   - 计算每个任务的最晚开始时间 (LS)
    │   - 计算每个任务的最晚结束时间 (LF)
    │
    └── 关键路径识别
        - ES = LS 的任务为关键任务
        - 关键任务组成的路径为关键路径

步骤4: 并行执行组划分
    │
    ├── 拓扑排序
    │   - 对DAG进行拓扑排序
    │   - 确定执行顺序
    │
    ├── 层级划分
    │   - 同一层级的任务无依赖
    │   - 可并行执行
    │
    └── 组生成
        - 每个层级为一个并行执行组
        - 标记组类型: 串行/并行
```

## 依赖类型识别规则

### 硬依赖识别

```yaml
hard_dependency_rules:
  - rule: 输出依赖
    condition: "任务B的输入需要任务A的输出"
    example: "T1(需求分析) → T2(API设计)"
    reason: "API设计需要需求文档作为输入"
    
  - rule: 文件依赖
    condition: "任务B需要修改任务A创建的文件"
    example: "T1(创建模型) → T2(创建控制器)"
    reason: "控制器依赖模型文件"
    
  - rule: 阶段依赖
    condition: "任务B属于任务A的后续阶段"
    example: "T1(开发) → T2(测试)"
    reason: "测试必须在开发完成后执行"
    
  - rule: 资源依赖
    condition: "任务B需要任务A释放的资源"
    example: "T1(数据库迁移) → T2(数据导入)"
    reason: "数据导入需要迁移完成"
```

### 软依赖识别

```yaml
soft_dependency_rules:
  - rule: 效率优化
    condition: "任务B在任务A后执行效率更高"
    example: "T1(代码分析) ⇢ T2(代码优化)"
    reason: "优化建议基于分析结果，但可独立执行"
    
  - rule: 资源竞争
    condition: "任务A和B操作相同资源，建议顺序执行"
    example: "T1(读取配置) ⇢ T2(修改配置)"
    reason: "避免读写冲突，但可接受"
    
  - rule: 信息共享
    condition: "任务B可利用任务A的结果提高质量"
    example: "T1(代码评审) ⇢ T2(代码重构)"
    reason: "重构可参考评审建议，但非必须"
```

### 无依赖识别

```yaml
no_dependency_rules:
  - rule: 文件隔离
    condition: "任务A和B操作不同的文件集"
    example: "T1(前端开发) , T2(后端开发)"
    reason: "文件范围无重叠，可并行"
    
  - rule: 模块隔离
    condition: "任务A和B操作不同的模块"
    example: "T1(用户模块) , T2(订单模块)"
    reason: "模块边界清晰，无依赖"
    
  - rule: 独立功能
    condition: "任务A和B实现独立的功能"
    example: "T1(日志分析) , T2(代码分析)"
    reason: "功能独立，无依赖"
```

## 依赖图构建

### 图结构定义

```yaml
DependencyGraph:
  nodes:
    - id: string              # 任务ID
      name: string            # 任务名称
      file_scope: string[]    # 文件范围
      complexity: enum        # 复杂度: low/medium/high
      duration: number        # 预估时长(分钟)
      
  edges:
    - from: string            # 起始任务ID
      to: string              # 目标任务ID
      type: enum              # 依赖类型: hard/soft
      reason: string          # 依赖原因
      
  properties:
    total_tasks: number       # 总任务数
    total_edges: number       # 总边数
    critical_path: string[]   # 关键路径
    max_parallelism: number   # 最大并行度
```

### 图构建示例


## 关键路径计算

### 计算方法

```yaml
critical_path_calculation:
  step1_forward_pass:
    description: "正向遍历计算最早时间"
    formula:
      ES[T1] = 0                           # 起始任务最早开始时间为0
      EF[T] = ES[T] + Duration[T]          # 最早结束时间
      ES[T] = max(EF[predecessors])        # 最早开始时间 = 前驱最早结束时间的最大值
      
  step2_backward_pass:
    description: "反向遍历计算最晚时间"
    formula:
      LF[T_last] = EF[T_last]              # 最后任务的最晚结束时间 = 最早结束时间
      LS[T] = LF[T] - Duration[T]          # 最晚开始时间
      LF[T] = min(LS[successors])          # 最晚结束时间 = 后继最晚开始时间的最小值
      
  step3_identify_critical:
    description: "识别关键任务"
    formula:
      critical_task: ES[T] == LS[T]        # 最早开始时间 = 最晚开始时间
```

### 计算示例

```

- T1: 10分钟
- T2: 15分钟
- T3: 20分钟
- T4: 15分钟

│ T7   │ 20   │ 60  │ 80  │ ES = max(EF[T5], EF[T6]) = 60     │
```

## 并行执行组划分

### 划分算法

```yaml
parallel_group_algorithm:
  step1_topological_sort:
    description: "对DAG进行拓扑排序"
    method: "Kahn算法或DFS"
    
  step2_level_assignment:
    description: "为每个任务分配层级"
    formula:
      level[T] = 0                              # 无前驱的任务层级为0
      level[T] = max(level[predecessors]) + 1   # 其他任务层级 = 前驱最大层级 + 1
      
  step3_group_generation:
    description: "按层级生成并行执行组"
    rule: "同层级的任务为同一并行执行组"
```

### 划分示例

```
层级划分:
┌─────────────────────────────────────────────────────────────┐
│ 层级 │ 任务      │ 说明                                     │
├──────┼───────────┼──────────────────────────────────────────┤
│ 0    │ T1        │ 无前驱，层级为0                          │
│ 1    │ T2, T3, T4│ 前驱为T1，层级为1                        │
│ 2    │ T5, T6    │ 前驱为T2/T3/T4，层级为2                  │
│ 3    │ T7        │ 前驱为T5/T6，层级为3                      │
└─────────────────────────────────────────────────────────────┘

并行执行组:
┌─────────────────────────────────────────────────────────────┐
│ 组号 │ 类型   │ 任务      │ 说明                            │
├──────┼────────┼───────────┼─────────────────────────────────┤
│ 组1  │ 串行   │ T1        │ 单任务，串行执行                 │
│ 组2  │ 并行   │ T2, T3, T4│ 3个任务，并行执行                │
│ 组3  │ 并行   │ T5, T6    │ 2个任务，并行执行                │
│ 组4  │ 串行   │ T7        │ 单任务，串行执行                 │
└─────────────────────────────────────────────────────────────┘

并行效率分析:
- 串行执行总时长: 10+15+20+15+30+25+20 = 135分钟
- 并行执行总时长: 80分钟
- 时间节省: (135-80)/135 = 40.7%
- 最大并行度: 3 (组2)
```

## 分析原则

| 原则 | 描述 | 强制性 |
|------|------|--------|
| **无环检测** | 确保依赖图无环 | ✅ 必须 |
| **关键路径优先** | 关键任务优先分配资源 | ✅ 必须 |
| **最大化并行** | 尽可能划分更多并行组 | ✅ 必须 |
| **文件隔离验证** | 确保并行任务文件范围不重叠 | ✅ 必须 |
| **软依赖处理** | 评估软依赖是否可并行 | ✅ 必须 |
