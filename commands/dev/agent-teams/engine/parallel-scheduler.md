# 并行调度器 (Parallel Scheduler)

并行调度器是并行执行引擎的第三个核心组件，负责调度和执行并行任务组，实现最大化并行。

---

## 调度算法

### 算法入口

```
输入: 并行执行组 (Parallel Execution Groups)
输出: 执行结果 (Execution Results)
```

### 调度步骤

```
步骤1: 执行组遍历
    │
    ├── 按顺序遍历每个执行组
    │   - 组1 → 组2 → 组3 → ...
    │
    └── 检查前置条件
        - 前一组必须完成
        - 质量门必须通过

步骤2: 任务分配
    │
    ├── 串行组
    │   - 单任务，直接执行
    │   - 使用 Task 工具启动单个 agent
    │
    └── 并行组
        - 多任务，并行执行
        - 使用 Task 工具同时启动多个 agent
        - 在单个消息中发起多个 Task 调用

步骤3: 执行监控
    │
    ├── 状态追踪
    │   - 跟踪每个 agent 的执行状态
    │   - 记录执行进度
    │
    ├── 冲突检测
    │   - 检测文件冲突
    │   - 检测资源冲突
    │
    └── 异常处理
        - 超时处理
        - 错误恢复
        - 任务重试

步骤4: 结果聚合
    │
    ├── 等待完成
    │   - 等待所有并行任务完成
    │   - 收集执行结果
    │
    ├── 结果汇总
    │   - 合并各 agent 的输出
    │   - 生成汇总报告
    │
    └── 质量验证
        - 执行质量检查
        - 通过后进入下一组
```

## 并行启动规范

### 核心原则

```yaml
parallel_launch_rules:
  rule1_simultaneous_launch:
    description: "同组任务必须在单个消息中同时启动"
    implementation: "在单个消息中发起多个 Task 工具调用"
    example: |
      ✅ 正确:
      [消息中同时包含 Task(T1), Task(T2), Task(T3)]
      
      ❌ 错误:
      [消息1: Task(T1)] → 等待完成 → [消息2: Task(T2)] → ...
      
  rule2_file_isolation:
    description: "并行任务的文件范围必须不重叠"
    implementation: "启动前检查文件范围"
    example: |
      T1 文件范围: src/frontend/ → 无冲突 ✅
      T2 文件范围: src/backend/ → 无冲突 ✅
      
  rule3_resource_isolation:
    description: "并行任务的资源使用必须不冲突"
    implementation: "分配不同的资源窗口"
    example: |
      T1: 读取数据库 → 无冲突 ✅
      T2: 写入文件系统 → 无冲突 ✅
      
  rule4_priority_scheduling:
    description: "关键路径任务优先调度"
    implementation: "关键任务标记为高优先级"
    example: |
      T1 (关键) → 优先分配资源
      T2 (非关键) → 正常分配资源
```

## Task 工具调用规范

### Agent 任务模板

```yaml
agent_task_template:
  subagent_type: "search"  # 或其他类型
  
  description: "[任务名称] - [简短描述]"
  
  query: |
    你是一个专门执行 [任务类型] 任务的 Agent。
    
    ## 任务目标
    [详细描述任务目标]
    
    ## 文件范围
    [列出需要操作的文件/目录]
    
    ## 输入
    [任务输入信息]
    
    ## 输出要求
    [描述期望的输出格式]
    
    ## 注意事项
    - 只操作指定文件范围内的文件
    - 不要修改其他 Agent 正在操作的文件
    - 完成后返回结构化的执行结果
    
  response_language: "中文"
```

### 并行调用示例

```
🚀 启动并行执行组2 (T2, T3, T4)

┌─────────────────────────────────────────────────────────────┐
│ 正在并行启动 3 个 Agent...                                   │

[同时调用 Task 工具启动 agent-1, agent-2, agent-3]
```

## 冲突检测与处理

### 冲突类型

```yaml
conflict_types:
  file_conflict:
    description: "多个 Agent 操作同一文件"
    detection: "检查文件范围是否重叠"
    resolution: "重新分配文件范围或串行执行"
    
  resource_conflict:
    description: "多个 Agent 竞争同一资源"
    detection: "检查资源锁状态"
    resolution: "分配不同的时间窗口"
    
  output_conflict:
    description: "多个 Agent 输出到同一位置"
    detection: "检查输出路径是否重叠"
    resolution: "分配不同的输出路径"
```

### 冲突检测流程

```
冲突检测流程:

步骤1: 文件范围检查
    │
    ├── 获取所有任务的文件范围
    │
    ├── 检查是否有重叠
    │   - 有重叠 → 标记冲突
    │   - 无重叠 → 通过检查
    │
    └── 输出检查结果

步骤2: 资源使用检查
    │
    ├── 获取所有任务的资源需求
    │
    ├── 检查是否有冲突
    │   - 有冲突 → 分配时间窗口
    │   - 无冲突 → 通过检查
    │
    └── 输出检查结果

步骤3: 输出路径检查
    │
    ├── 获取所有任务的输出路径
    │
    ├── 检查是否有重叠
    │   - 有重叠 → 分配不同路径
    │   - 无重叠 → 通过检查
    │
    └── 输出检查结果
```

### 冲突检测模板

```markdown
🔍 冲突检测 - 组[N]

文件范围检查:
- T2: docs/api/ → 无冲突 ✅
- T3: docs/db/ → 无冲突 ✅
- T4: docs/ui/ → 无冲突 ✅

资源使用检查:
- T2: 读取 package.json → 无冲突 ✅
- T3: 读取 package.json → 无冲突 ✅
- T4: 无资源依赖 → 无冲突 ✅

输出路径检查:
- T2: docs/api/login-api.md → 无冲突 ✅
- T3: docs/db/user-table.md → 无冲突 ✅
- T4: docs/ui/login-ui.md → 无冲突 ✅

结论: 可以并行执行 ✅
```

## 异常处理

### 异常类型

```yaml
exception_types:
  timeout:
    description: "任务执行超时"
    threshold: "根据任务复杂度设定"
    handling:
      - 记录超时信息
      - 尝试重新执行
      - 如多次失败，上报用户
      
  error:
    description: "任务执行错误"
    handling:
      - 记录错误详情
      - 分析错误原因
      - 尝试修复或跳过
      
  resource_unavailable:
    description: "资源不可用"
    handling:
      - 等待资源释放
      - 尝试使用替代资源
      - 如无法获取，上报用户
      
  dependency_failed:
    description: "依赖任务失败"
    handling:
      - 等待依赖任务修复
      - 重新评估执行计划
      - 可能需要重新分解任务
```

### 异常处理流程

```
异常处理流程:

检测到异常
    │
    ├── 记录异常信息
    │   - 异常类型
    │   - 异常详情
    │   - 影响范围
    │
    ├── 分析异常原因
    │   - 代码错误
    │   - 资源问题
    │   - 依赖问题
    │
    ├── 尝试恢复
    │   ├── 可恢复 → 重新执行
    │   └── 不可恢复 → 上报用户
    │
    └── 更新执行状态
        - 更新任务状态
        - 更新依赖关系
        - 重新计算执行计划
```

## 结果聚合

### 聚合流程

```yaml
result_aggregation:
  step1_collect:
    description: "收集所有 Agent 的执行结果"
    actions:
      - 等待所有并行任务完成
      - 获取每个任务的输出
      
  step2_validate:
    description: "验证结果完整性"
    checks:
      - 检查输出格式是否正确
      - 检查输出内容是否完整
      - 检查是否有遗漏
      
  step3_merge:
    description: "合并结果"
    actions:
      - 合并相同类型的输出
      - 建立输出间的关联关系
      - 生成汇总报告
      
  step4_quality_check:
    description: "执行质量检查"
    checks:
      - 代码编译检查
      - 类型检查
      - 规范检查
      - 测试检查
```

## 调度原则

| 原则 | 描述 | 强制性 |
|------|------|--------|
| **同时启动** | 同组任务必须在单个消息中同时启动 | ✅ 必须 |
| **文件隔离** | 并行任务文件范围不重叠 | ✅ 必须 |
| **状态追踪** | 实时追踪每个任务的执行状态 | ✅ 必须 |
| **异常处理** | 及时处理异常，不阻塞其他任务 | ✅ 必须 |
| **结果聚合** | 等待所有任务完成后聚合结果 | ✅ 必须 |
| **质量验证** | 每组完成后执行质量检查 | ✅ 必须 |
