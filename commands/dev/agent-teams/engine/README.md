## 核心模块

### 1. 任务分解器 (Task Decomposer)

**文件**: [task-decomposer.md](task-decomposer.md)

**功能**:
- 识别任务类型
- 将用户请求分解为可并行执行的子任务
- 确保每个子任务有明确的输入、输出和文件范围

**输入**: 用户请求（自然语言）

**输出**: 任务分解表

**支持的任务类型**:
| 类型 | 触发关键词 | 分解维度 |
|------|-----------|---------|
| 探索类 | 探索、分析、了解 | 文件/目录范围 |
| 开发类 | 开发、实现、新增 | 执行阶段 + 文件范围 |
| 修复类 | 修复、解决、bug | 分析维度 + 执行阶段 |
| 评审类 | 评审、检查、审查 | 评审维度 |
| 部署类 | 部署、发布、上线 | 部署阶段 |

---

### 2. 依赖分析器 (Dependency Analyzer)

**文件**: [dependency-analyzer.md](dependency-analyzer.md)

**功能**:
- 分析任务间的依赖关系
- 构建依赖图 (DAG)
- 计算关键路径
- 划分并行执行组

**输入**: 任务分解表

**输出**: 依赖图 + 并行执行组

**依赖类型**:
| 类型 | 描述 | 处理方式 |
|------|------|---------|
| 硬依赖 | B必须在A完成后才能开始 | 顺序执行 |
| 软依赖 | B建议在A后执行，但可并行 | 评估后决定 |
| 无依赖 | A和B完全独立 | 并行执行 |

---

### 3. 并行调度器 (Parallel Scheduler)

**文件**: [parallel-scheduler.md](parallel-scheduler.md)

**功能**:
- 调度和执行并行任务组
- 实现最大化并行
- 监控执行状态
- 处理异常情况
- 聚合执行结果

**输入**: 并行执行组

**输出**: 执行结果

**核心原则**:
| 原则 | 描述 |
|------|------|
| 同时启动 | 同组任务必须在单个消息中同时启动 |
| 文件隔离 | 并行任务文件范围不重叠 |
| 状态追踪 | 实时追踪每个任务的执行状态 |
| 异常处理 | 及时处理异常，不阻塞其他任务 |

## 执行流程

```
步骤1: 任务分解
    │
    ├── 识别任务类型
    ├── 按维度分解任务
    └── 生成任务分解表
    
步骤2: 依赖分析
    │
    ├── 识别依赖类型
    ├── 构建依赖图
    ├── 计算关键路径
    └── 划分并行执行组
    
步骤3: 并行调度
    │
    ├── 遍历执行组
    ├── 同时启动同组任务
    ├── 监控执行状态
    └── 聚合执行结果
    
步骤4: 质量验证
    │
    ├── 执行质量检查
    └── 通过后进入下一阶段
```

## 配置模板

配置模板位于 `../config/templates/` 目录，每个模板包含：

```yaml
execution_engine:
  enabled: true           # 启用执行引擎
  auto_decompose: true    # 自动任务分解
  auto_analyze: true      # 自动依赖分析
  auto_schedule: true     # 自动并行调度

decomposition:
  trigger_keywords: [...] # 触发关键词
  rules: [...]            # 分解规则

dependency_analysis:
  dependencies: [...]     # 依赖关系定义
  parallel_groups: [...]  # 并行执行组定义

parallel_scheduling:
  max_parallelism: N      # 最大并行度
  launch_rule: simultaneous  # 同时启动
  conflict_detection: true   # 冲突检测
  quality_gates: [...]    # 质量门定义
```

## 使用示例

### 探索代码库

```
用户请求: "探索这个代码库的架构"

执行流程:
1. 任务分解 → 生成8个探索任务（目录、核心模块、依赖、配置等）
2. 依赖分析 → 所有任务无依赖，形成1个并行组
3. 并行调度 → 同时启动8个Agent探索不同范围
4. 结果聚合 → 生成探索报告

时间节省: 约85%（串行需要逐个探索，并行同时探索）
```

### 开发功能

```
用户请求: "开发用户登录功能"

执行流程:
1. 任务分解 → 生成7个任务（需求、设计、开发、测试）
2. 依赖分析 → 形成4个执行组
   - 组1: 需求分析（串行）
   - 组2: API设计、数据库设计、UI设计（并行）
   - 组3: 前端开发、后端开发（并行）
   - 组4: 单元测试（串行）
3. 并行调度 → 按组顺序执行，组内并行
4. 结果聚合 → 生成开发报告

时间节省: 约40%
```

## 核心原则

| 原则 | 描述 | 强制性 |
|------|------|--------|
| **并行优先** | 最大化并行，最短时间完成 | ✅ 必须 |
| **任务分解** | 所有任务必须分解到可并行粒度 | ✅ 必须 |
| **文件隔离** | 不同Agent操作不同文件集 | ✅ 必须 |
| **质量优先** | 质量是底线，必须通过 | ✅ 必须 |
| **状态同步** | 实时监控，及时处理异常 | ✅ 必须 |
